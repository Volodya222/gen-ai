

## 1. Инструмент compare_periods

**Сигнатура** (из schemas.py):
```python
compare_periods(
    metric: Literal["fx_USD", "fx_EUR", "key_rate", "inflation", "unemployment"],
    period_a: str,  # формат "YYYY-MM"
    period_b: str,  # формат "YYYY-MM"
) -> CompareResult
```

`CompareResult` содержит: `metric`, `a` (дата + значение), `b` (дата + значение),
`delta` (b − a), `ratio` (b / a).

**Примеры использования агентом:**

*Q5 — «Во сколько раз вырос курс USD с января 2022 по апрель 2026?»*
```
compare_periods({'metric': 'fx_USD', 'period_a': '2022-01', 'period_b': '2026-04'})
→ {"a": {"date": "2022-01-01", "value": 74.2926},
   "b": {"date": "2026-04-01", "value": 81.2504},
   "delta": 6.9578, "ratio": 1.0937}
Ответ: курс вырос в 1.09 раза.
```

*Q6 — «На сколько изменилась ключевая ставка с января 2022 по март 2026?»*
```
compare_periods({'metric': 'key_rate', 'period_a': '2022-01', 'period_b': '2026-03'})
→ {"a": {"date": "2022-01-01", "value": 8.5},
   "b": {"date": "2026-03-01", "value": 17.0},
   "delta": 8.5, "ratio": 2.0}
Ответ: ставка выросла на 8.5 п.п. (удвоилась).
```

## 3. Диагностика трёх типов ошибок

**Тип 1 — пропуск инструмента (Q1: «Какая сегодня ключевая ставка ЦБ?»)**
Агент вообще не вызвал `get_key_rate` и сразу выдал финальный текст с пустым
ответом. Vодель решила, что знает ответ из обучающих данных, и не стала
обращаться к инструменту. Характерно для простых на вид вопросов: «ключевая
ставка» — частый запрос, и модель «помнит» какое-то число без проверки.
Исправление: добавить в системный промпт явный запрет отвечать на вопросы
о текущих данных без вызова инструмента.

**Тип 2 — данные недоступны, агент не пробует fallback (Q3, Q7)**
Q3: агент правильно вызвал `get_key_rate` (получил 16.0%) и `get_inflation`
(получил ошибку «нет данных ИПЦ на 2026-05»), но остановился — не попробовал
более ранний месяц и не вызвал `calculate` с последними известными данными.
Q7: аналогично — оба инструмента вернули ошибку на июнь 2026, агент сдался
вместо того чтобы попробовать май или апрель. Это хрупкость к граничным датам:
агент не умеет делать retry с другими параметрами при ошибке инструмента.

**Тип 3 — неверный порядок вызовов, исправленный retry (Q4)**
Первый шаг: агент попытался вызвать `calculate({'expression': '72 / str(get_key_rate...}')`
— попытка вложить вызов инструмента внутрь другого. Провалилось с ошибкой
«unexpected keyword argument». Второй шаг: агент скорректировался, вызвал
`get_key_rate` отдельно, получил 16.0, затем `calculate({'expression': '72 / 16'})`.
Итог: PASS. Позитивный пример: ReAct-цикл позволяет агенту исправить себя
после ошибки — без retry вопрос бы упал.

## 4. Вывод про хрупкость системы

Агент демонстрирует три системных слабости:

**Хрупкость к свежим данным** 

**Хрупкость к простым вопросам** 

**Устойчивость к ошибкам формата** 

Самой большой уязвимостью является поведение агента после возвращения ошибки. Агент сообщает, что данных нет вместо того, чтобы поискать в соседних периодах. Ошибка данных не является сигналом к тому, чтобы попробовать иначе.
